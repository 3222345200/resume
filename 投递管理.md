## 2. 模块定位

投递管理用于记录用户的每一次岗位申请，并持续维护其状态变化、跟进动作和关联简历。

它主要解决三个问题：

1. 管理机会：记录每一次岗位申请及其进展
2. 管理过程：追踪从投递到面试再到 Offer 的整个流程
3. 管理执行：明确哪些记录需要跟进、下一步要做什么

在整个产品中的关系如下：

- 简历模块负责管理“投递材料”
- 投递管理负责管理“机会与状态”
- 面试记录负责管理“过程与复盘”

三者共同构成完整的求职闭环。

## 3. 核心使用流程

1. 用户创建一条投递记录
2. 填写公司、岗位、渠道、投递日期等信息
3. 选择本次投递使用的简历
4. 随着求职进展持续更新投递状态
5. 若收到面试邀约，则在该投递下新增面试记录
6. 最终记录结果为 Offer、拒绝、无反馈或主动结束

## 4. 设计原则

### 4.1 功能范围原则

投递管理模块的功能范围应严格围绕“岗位申请记录管理”展开，不额外混入简历编辑或面试复盘本身的主流程。

### 4.2 与现有简历模块统一

投递管理模块在视觉风格、布局逻辑和交互节奏上，应尽量继承当前已上线的“简历管理 / 简历编辑器”模块，而不是重新设计为另一套完全不同的后台页面。

建议统一以下原则：

- 功能内容以投递管理定义为准
- 信息结构延续现有工作台式布局
- 视觉语言延续现有简历管理页面的浅色 SaaS 风格
- 交互方式优先采用“列表联动详情”而非频繁弹窗
- 用户在“简历管理”和“投递管理”之间切换时，保持熟悉的页面节奏和操作心智

## 5. 页面目标

投递管理页用于帮助用户：

- 快速查看当前所有岗位申请
- 了解每条投递目前处于什么阶段
- 查找需要跟进的机会
- 进入单条投递详情查看完整信息

## 6. 页面结构

建议采用与现有简历管理模块一致的工作台式三栏结构：

- 左侧：工作台导航 / 快捷视图区
- 中间：投递概览、筛选操作、投递列表主区域
- 右侧：当前投递的详情联动区

### 6.1 左侧工作台侧栏 / 快捷视图区

建议包含：

- 产品标识与工作台名称
- 用户信息
- 工作台入口导航：工作台、简历管理、投递管理、面试记录
- 投递管理下的快捷视图：全部投递、待跟进、面试中、Offer、已拒绝、本周新增
- 返回工作台、退出登录等次级操作

### 6.2 顶部概览区

用于展示求职整体状态，建议包含：

- 总投递数
- 本周新增投递数
- 面试中数量
- Offer 数量
- 已拒数量
- 超过一定时间未反馈数量

### 6.3 筛选与操作区

用于快速缩小查看范围并触发新增操作，建议包含：

- 关键词搜索：公司名、岗位名
- 状态筛选
- 渠道筛选
- 城市 / 地点筛选
- 时间范围筛选
- “仅看待跟进”筛选
- 新建投递按钮

### 6.4 投递列表区

用于展示所有投递记录，建议每条记录展示：

- 公司名
- 岗位名
- 工作地点
- 当前状态
- 投递日期
- 投递渠道
- 关联简历
- 最近更新时间
- 是否待跟进

建议补充以下交互要求：

- 点击某条记录后，右侧详情区同步切换
- 当前选中记录需要有明确高亮态
- 状态支持在列表中快速切换
- 待跟进记录在列表中需要有轻量提醒

### 6.5 投递详情区

点击列表项后展示详情，建议包含以下分区：

#### 1. 基本信息

- 岗位链接

#### 2. 投递信息

- 投递日期
- 投递渠道
- 内推人 / 联系人
- 联系方式

#### 3. 状态流转

- 当前状态
- 历史状态变更记录
- 最近一次跟进时间
- 下次跟进时间
- 截止日期

#### 4. 简历关联

## 8. 投递状态设计

建议先采用一套简单、覆盖主流程的状态体系：

- 想投
- 已投递
- 笔试
- 面试中
- HR 面
- Offer
- 已拒绝
- 已结束

- 一面

## 15. 校验与业务规则建议

- 所有投递记录必须绑定当前登录用户
- `resume_id` 必须属于当前用户
- 删除简历时，如存在关联投递，需要明确策略
- `next_follow_up_at` 允许为空
- `status` 必须受限于统一枚举
- 进入“Offer”或“已拒绝”后，可自动设置 `final_result`
- “待跟进”可根据 `next_follow_up_at <= 当前时间` 自动计算，也可冗余存储 `is_todo`

## 16. 与面试记录的联动规则建议

- 新增面试记录后，可提示将投递状态更新为“面试中”
- 面试通过后，可提示进入下一轮或更新为更细的面试阶段
- 面试淘汰后，可提示更新投递状态为“已拒绝”
- 投递详情页应显示当前已关联的面试轮次数
- 投递详情页应能直接跳转或创建面试记录

## 17. 实施建议

建议按以下顺序落地：

1. 数据模型：`applications` 主表 + 基础 CRUD
2. 列表接口：支持搜索、筛选、概览统计
3. 详情接口：补状态历史与关联信息
4. 快速状态更新与跟进更新接口
5. 前端工作台页面：列表 + 右侧详情联动
6. 与面试记录模块做基础联动

【产品背景】

OfferPilot 是一个求职工作台产品。当前已经有成熟的「简历管理 / 在线简历编辑器」模块，支持多份简历管理、在线编辑、模板选择、实时预览和 PDF 导出。

现在要新增「投递管理」模块，用于记录每一次岗位申请，管理状态变化、跟进计划、关联简历和面试记录。


【重要要求】

这个新页面的功能内容必须遵循“投递管理”产品设计，视觉风格和交互节奏必须继承当前已有的“简历管理模块”。

也就是说：

- 功能上是投递管理

- 风格上要像同一个产品里的简历管理页面

- 不要设计成另一套完全不同的后台系统


【页面目标】

这个页面要帮助用户：

- 快速查看当前所有岗位申请

- 了解每条投递目前处于什么阶段

- 快速找到需要跟进的机会

- 点击查看单条投递的完整详情

- 快速新建投递、筛选记录、更新状态

- 查看关联简历和面试记录入口


【页面类型】

桌面端 Web SaaS 后台页面

宽屏布局，参考 1440px 到 1600px

浅色主题

高保真 UI

真实产品感

信息密度适中，强调结构清晰和高频使用体验


【整体布局】

请采用与当前简历管理模块一致的“三栏工作台结构”：


1. 左侧：工作台侧栏 / 快捷视图区

2. 中间：投递管理主区域

3. 右侧：投递详情联动区


整体要让人一眼感觉这是和“简历管理模块”同一产品、同一设计体系下的页面。


【A. 左侧工作台侧栏】

左侧侧栏延续简历管理模块的风格，包含：

- OfferPilot Logo 和产品名

- 一句简短描述，例如：从简历制作到投递跟进的求职工作台

- 当前登录用户信息

- 工作台导航入口：

  - 工作台

  - 简历管理

  - 投递管理（当前选中）

  - 面试记录

- 投递快捷视图：

  - 全部投递

  - 待跟进

  - 面试中

  - Offer

  - 已拒绝

  - 本周新增

- 底部次级操作：

  - 返回工作台

  - 退出登录


要求这个侧栏的气质要和现有简历管理模块侧栏保持一致，简洁、克制、专业。


【B. 中间主工作区】

中间为投递管理主体，分成三层：


第一层：顶部概览区

使用概览卡片展示关键数据，包含：

- 总投递数

- 本周新增投递数

- 面试中数量

- Offer 数量

- 已拒数量

- 超过一定时间未反馈数量


要求：

- 卡片简洁、数字突出

- 风格要像成熟的效率工具后台

- 不要夸张视觉效果


第二层：筛选与操作区

放在概览卡片下方，作为一整条工具栏，包含：

- 搜索框：支持公司名、岗位名搜索

- 状态筛选

- 渠道筛选

- 城市 / 地点筛选

- 时间范围筛选

- “仅看待跟进”开关

- 右侧主按钮：“新建投递”


要求：

- 工具栏布局清晰紧凑

- 交互控件真实可用

- 新建投递按钮要明显突出


第三层：投递列表区

用于展示所有投递记录，每条记录展示：

- 公司名

- 岗位名

- 工作地点

- 当前状态

- 投递日期

- 投递渠道

- 关联简历

- 最近更新时间

- 是否待跟进


列表要求：

- 不要做成死板传统表格

- 更像高信息密度的 SaaS 列表卡片 / 记录列表

- 当前选中记录高亮

- 点击某条记录后，右侧详情区同步切换

- 每条记录右侧可有快捷操作：编辑、更新状态、更多操作


状态标签颜色建议：

- 想投：灰色

- 已投递：蓝色

- 笔试：紫蓝色

- 面试中：橙色

- HR 面：青色

- Offer：绿色

- 已拒绝：红色

- 已结束：深灰色


“待跟进”记录要求：

- 有轻量但明显的提醒方式

- 可以用小圆点、浅黄色标签、浅色背景高亮

- 不要用过于刺眼的警报式设计


【C. 右侧投递详情联动区】

右侧详情区的定位，要对应简历管理模块里的“实时预览区”，但这里展示的是“投递详情”。

要求采用固定右侧详情栏或抽屉式固定面板。


详情区按模块分组展示，包含：


1. 基本信息

- 公司名称

- 岗位名称

- 部门

- 工作地点

- 岗位链接

- JD 摘要

- 薪资范围

- 用工类型（校招 / 实习 / 社招）


2. 投递信息

- 投递日期

- 投递渠道

- 内推人 / 联系人

- 联系方式

- 备注


3. 状态流转

- 当前状态

- 历史状态变更记录

- 最近一次跟进时间

- 下次跟进时间

- 截止日期

- 下一步动作


4. 简历关联

- 本次投递使用的简历

- 简历标题或版本标识

- 查看简历 / 编辑简历快捷入口


5. 面试记录入口

- 当前已关联的面试轮次数

- 快速新增面试按钮

- 查看完整面试时间线入口


要求：

- 详情区采用清晰的模块分组卡片

- 风格要和简历管理模块里的编辑分区一致

- 强调信息层级和阅读效率


【交互状态要求】

请在静态设计稿里体现以下状态：

- 左侧“投递管理”导航已选中

- 中间列表有一条投递记录被选中

- 右侧显示该条投递的完整详情

- 页面中存在至少 1 条“待跟进”记录并被高亮

- 筛选栏有已选条件的感觉

- “新建投递”按钮可点击感明显

- 状态标签有明确视觉区分

- 整体看起来像真实可用产品，而不是概念稿


【视觉风格】

整体风格要求：

- 现代、克制、专业、可信赖

- 偏 SaaS、效率工具、工作台

- 与当前简历管理模块风格统一

- 类似 Notion、Linear、Airtable 的精致后台感，但不要照搬

- 不是营销官网，不要插画主视觉

- 不要玻璃拟态，不要大面积炫彩渐变

- 强调卡片层次、信息组织、长时间使用舒适度


【配色建议】

- 主色：深蓝或青蓝色

- 背景：白色 / 浅灰白

- 卡片：浅阴影或浅边框

- 文本：深灰

- 分割线：浅灰

- 成功：绿色

- 警示：橙黄

- 风险 / 拒绝：红色

- 保持整体克制、成熟、统一


【文案要求】

- 所有界面文案使用中文

- 数据模拟要真实可信，符合求职场景

- 可使用真实风格示例公司，如：

  - 字节跳动

  - 阿里云

  - 腾讯

  - Shopee

  - 小红书

- 示例岗位可为：

  - 后端开发工程师

  - 产品经理

  - 数据分析师

  - 前端开发工程师


【输出要求】

请直接输出：

一张完整的高保真桌面端「投递管理」页面设计图


不要输出：

- 纯文字分析

- 线框图

- 手机端界面

- 分散的多个小页面

- 海报式视觉稿


重点是：

这是一个真实能用的求职管理后台页面，并且必须与已有的“简历管理模块”保持同一套设计语言和工作台体验。
